# 2026-07-14 — Milestone 4.2: HTTP middleware (Logger)

## What got done

- **Roadmap 4.2 `middleware.Logger`** (branch `feat/middleware-logger`, ADR-0014,
  patterns row 9): the second HTTP middleware. Spec §5 surface —
  `Logger(l *slog.Logger) func(http.Handler) http.Handler` — emits exactly one structured
  `log/slog` line per request after the downstream handler returns, carrying method, path,
  status, duration, and bytes-written, plus `request_id` when the chain seeded one (4.1).
- **Status and byte capture without breaking the `ResponseWriter` contract.** A
  `responseRecorder` embeds the real writer, records the first `WriteHeader` code
  (defaulting to 200) and accumulates `Write` bytes, and exposes `Unwrap()
  http.ResponseWriter` so `http.ResponseController` (Go 1.20+) still reaches the underlying
  Flusher / Hijacker / ReaderFrom — verified by `TestLoggerPreservesFlusherThroughUnwrap`.
- **Level follows status** (5xx→Error, 4xx→Warn, else Info), so operators can query the
  5xx at the source; overridable by the consumer's `slog.Handler`.
- **Security decision — path only, never the query string** (and no headers, no body).
  Query parameters routinely carry secrets; logging them is a classic information-disclosure
  leak. This is why the item carries an ADR under the enterprise posture: it **extends** the
  threat model's Info-disclosure row (a Logger row) and **compliance control C-2** rather
  than opening new artifacts. Logged from a `defer`, so a panicking request is still logged
  before the panic propagates; the recommended chain order is
  `RequestID → Logger → Recoverer → handler` (documented in the godoc + ADR-0014).
- Local gauntlet green (portable Go 1.26.5): `go build ./...`, `go vet ./middleware/`, full
  `go test ./...`, gofumpt clean. Note: `-race` needs cgo and there is no C compiler in the
  local box, so the race detector runs in CI only (as with prior milestones).
